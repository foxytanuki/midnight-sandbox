# 証明ライフサイクル — Midnightでの全体の流れ

[← Trusted Setup](./80-06-trusted-setup.md) | [全体像に戻る →](./80-00-zk-overview.md)

## 概要

Midnightでゼロ知識証明が生成・検証されるまでの全工程を追う。

```
┌─────────────┐
│  開発者      │ Compact コードを書く
└──────┬──────┘
       ↓ コンパイル
┌─────────────┐
│  回路 + キー │ 証明キー (pk) と検証キー (vk) が生成される
└──────┬──────┘
       ↓ デプロイ
┌─────────────┐
│  チェーン    │ 検証キーがオンチェーンに登録される
└──────┬──────┘
       │
       │  ユーザーがdAppを使うとき:
       ↓
┌─────────────┐
│  Proof Server│ ローカルで証明を生成
└──────┬──────┘
       ↓ トランザクション送信
┌─────────────┐
│  バリデータ  │ オンチェーンで証明を検証
└─────────────┘
```

## Phase 1: コンパイル

### 入力

Compact言語で書かれたスマートコントラクト:

```typescript
// 概念的な例（簡略化）
circuit loan_check(
  secret credit_score: Field,
  secret monthly_income: Field,
  public threshold: Field
): Bool {
  let qualifies = credit_score >= 700 && monthly_income >= threshold;
  return qualifies;
}
```

### 処理

1. **パース・型チェック**: Compact コードを解析
2. **IR変換**: 中間表現に変換
3. **回路生成**: Plonkのゲート制約に変換
   - 比較演算 → ビット分解 + 範囲チェック
   - 論理積 → 乗算ゲート
   - 各変数 → ワイヤー
4. **最適化**: 不要なゲートの除去、カスタムゲートの適用
5. **キー生成**:
   - **証明キー (pk)**: 証明生成に必要（SRS + 回路固有の前処理データ）
   - **検証キー (vk)**: 検証に必要（証明キーより小さい）

### 出力

- 回路定義（制約の集合）
- 証明キー（pk）
- 検証キー（vk）

証明キーはProof Serverに、検証キーはチェーン上にデプロイされる。

## Phase 2: データ取得（Attestation）

ユーザーがdAppを使うとき、まず秘密データの取得と検証が必要。

### Attestation の流れ

```
┌──────────┐    API     ┌───────────────┐
│ dAppの    │ ←------→  │ データ        │
│ バックエンド│           │ プロバイダ     │
└────┬─────┘           │ (銀行, 信用調査│
     │                  │  機関等)       │
     │                  └───────┬───────┘
     │                          │
     │                   データ + 署名
     │                          │
     ↓                          ↓
┌──────────────────────────────────┐
│  ユーザーの端末                    │
│  秘密データ + プロバイダの署名      │
└──────────────────────────────────┘
```

- プロバイダはデータにデジタル署名する
- 署名はZK回路内で検証される → 署名と異なるデータでは証明を作れない
- データ自体はユーザーの端末から外に出ない

## Phase 3: 証明生成

### Proof Server

ユーザーのローカルマシンで動作するサーバー。

処理の流れ:

1. **Witness計算**: 秘密入力と公開入力から、回路の全ワイヤーの値を計算
2. **多項式構築**: Witnessの値をラグランジュ補間で多項式に変換
   - a(x), b(x), c(x): ワイヤー多項式
   - z(x): permutation argument の累積多項式
3. **商多項式計算**: ゲート制約・コピー制約の多項式を消失多項式で割る
4. **コミットメント生成**: 各多項式をKZGでコミット
5. **評価・Opening Proof**: Fiat-Shamirで生成したチャレンジ点での評価値と証明を計算

### 計算コスト

| 処理 | 主な演算 | 時間の目安 |
|---|---|---|
| Witness計算 | 有限体の算術演算 | ミリ秒〜秒 |
| FFT（多項式変換） | O(n log n) のFFT | 秒 |
| MSM（コミットメント） | n回のスカラー倍の和 | 秒〜十数秒 |
| 合計 | | 数秒〜数十秒 |

ボトルネックは**MSM（Multi-Scalar Multiplication）**: SRSの点にwitness値を掛けて足し合わせる処理。ゲート数に比例して重くなる。

### メモリ使用量

ゲート数 n に対して:
- Witness: O(n) の有限体要素（各32バイト）
- FFT用バッファ: O(n) の追加メモリ
- 10万ゲートの回路で数百MB程度

これがモバイル対応が課題である理由。

## Phase 4: トランザクション送信

生成された証明はトランザクションに含めてチェーンに送信:

```
トランザクション = {
  証明データ (π):     ~800バイト
  公開入力:          可変（承認結果など）
  コントラクトアドレス: 32バイト
  その他メタデータ:   署名、nonce等
}
```

秘密入力（年収、信用スコア等）はトランザクションに**含まれない**。

## Phase 5: オンチェーン検証

### バリデータの処理

1. トランザクションからπ（証明）と公開入力を取り出す
2. 検証キー（vk、事前にデプロイ済み）を読み込む
3. ペアリング計算を実行:
   ```
   e(A, B) = e(C, D)  // 2回のペアリング計算
   ```
4. 等式が成立すれば証明は有効 → トランザクションを承認
5. 不成立なら無効 → トランザクションを拒否

### 検証コスト

| 処理 | 時間 |
|---|---|
| 公開入力の処理 | マイクロ秒 |
| ペアリング計算（2回） | 数ミリ秒 |
| 合計 | 数ミリ秒 |

回路が1万ゲートでも100万ゲートでも検証時間はほぼ同じ。この非対称性（証明: 秒〜分、検証: ミリ秒）がZKの実用性を支える。

## Phase 6: 状態更新

検証が通ると:

1. コントラクトの状態が更新される（例: ローン承認フラグが立つ）
2. イベントがemitされる（例: `LoanApproved(applicantId)` ）
3. 公開された情報のみがチェーン上に記録される
4. 秘密情報はどこにも残らない

## セキュリティの境界

各フェーズで誰が何を信頼するか:

| フェーズ | 信頼の対象 | リスク |
|---|---|---|
| コンパイル | Compactコンパイラ | バグで不正な回路が生成される |
| データ取得 | データプロバイダ | 嘘のデータに署名される |
| 証明生成 | ローカル環境 | マルウェアがwitnessを盗む |
| オンチェーン検証 | 暗号学的仮定 + SRS | 量子コンピュータ、SRS漏洩 |
| 状態更新 | コンセンサス | 51%攻撃 |

ZKは「計算の正しさ」を保証するが、入力データの真正性（プロバイダの誠実さ）やユーザー環境のセキュリティまでは保証しない。
