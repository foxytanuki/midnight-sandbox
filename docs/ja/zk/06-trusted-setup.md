# Trusted Setup — 信頼されたセットアップ

[← 楕円曲線 BLS12-381](./05-bls12-381.md) | [次: 証明ライフサイクル →](./07-proof-lifecycle.md)

## なぜTrusted Setupが必要か

KZGコミットメントには**SRS（Structured Reference String）**と呼ばれる公開パラメータが必要:

```
SRS = [G₁, τG₁, τ²G₁, ..., τⁿG₁,  G₂, τG₂]
```

τ（tau）は秘密の乱数。SRSの生成後、τ自体は**絶対に破棄しなければならない**。

## τが漏洩するとどうなるか

τを知っている攻撃者は:

1. **任意の多項式のコミットメントを偽造できる**: f(τ) を直接計算してコミットメントを作れる
2. **偽の証明を作成できる**: 制約を満たさないwitnessに対しても有効な証明を生成できる
3. **健全性が完全に崩壊する**: ZK証明が何の保証もしなくなる

逆に、τを誰も知らなければ上記は計算上不可能。

## Powers of Tau セレモニー

### 基本アイデア

1人でτを生成すると、その1人を信頼しなければならない。代わりに**多数の参加者が順番にランダム性を加える**:

```
参加者1: τ₁ を選び、[G, τ₁G, τ₁²G, ...] を計算
参加者2: τ₂ を選び、[G, (τ₁τ₂)G, (τ₁τ₂)²G, ...] を計算
参加者3: τ₃ を選び、[G, (τ₁τ₂τ₃)G, (τ₁τ₂τ₃)²G, ...] を計算
...
```

最終的なτ = τ₁ · τ₂ · τ₃ · ... · τₖ

### セキュリティの保証

**1-of-N honest assumption（N人中1人が正直なら安全）**:

- 全参加者のうち1人でも自分のτᵢを正しく破棄すれば、最終的なτは誰にもわからない
- τ全体を復元するには全参加者のτᵢが必要
- 参加者1人のτᵢは他の参加者にも見えない

つまり、数千人の参加者のうち「全員が結託して秘密を保持した」場合のみ危険。現実的にはほぼ不可能。

### 手順の詳細

各参加者が行う処理:

1. 前の参加者からSRSを受け取る
2. ローカルでランダムなτᵢを生成
3. SRSの各要素にτᵢを「掛ける」:
   ```
   τⁿG → (τᵢ·τ)ⁿG = τᵢⁿ · τⁿ · G
   ```
4. 更新したSRSを次の参加者に渡す
5. τᵢを破棄する（メモリを消去、乱数生成器を破壊など）

### 検証

各参加者の更新が正しく行われたかは、ペアリングで検証できる:

```
e(τⁱG₁, G₂) = e(τⁱ⁻¹G₁, τG₂)
```

これにより、参加者が「何もしなかった」や「不正な値を入れた」ことを検出できる。

## 実績のあるセレモニー

### Zcash Powers of Tau（2017-2018）

- 87人の参加者
- 各参加者がVPSや専用ハードウェアで実行
- 一部の参加者は物理的にハードウェアを破壊して証拠を残した

### Ethereum KZG Ceremony（2023）

- **141,416人**が参加
- ブラウザから参加可能なWebアプリを用意
- 特殊なエントロピー源を使った参加者もいる（溶岩ランプの映像、大気ノイズ等）
- 1人でも正直に乱数を破棄していれば安全

### Midnightのセレモニー

Midnightも独自のPowers of Tauセレモニーを実施。具体的な参加者数や手順はMidnightの公式ドキュメントを参照。

## Universal Setup vs Circuit-Specific Setup

| | Circuit-Specific (Groth16) | Universal (Plonk) |
|---|---|---|
| セレモニー | 回路ごとに必要 | 1回でよい |
| SRSサイズ | 回路に依存 | 最大回路サイズに依存 |
| 回路変更時 | 再セレモニー | そのまま使える |
| 開発体験 | 面倒 | 良い |

Plonk（Universal Setup）は「SRSの最大次数 ≥ 回路のゲート数」であれば、どんな回路にも使い回せる。開発中に回路が変わってもセレモニーをやり直す必要がない。

## Trusted Setupへの批判と代替

### 批判

- 「信頼」が必要な時点でtrustlessの理念に反する
- セレモニーの運営自体への信頼も必要
- 長期的に量子コンピュータでτが復元される可能性

### 代替: Transparent Setup（STARKs）

STARKはTrusted Setupを必要としない。ハッシュ関数のみに依存する。

トレードオフ:
- 証明サイズが大きくなる（数十KB vs 数百バイト）
- 検証時間が長くなる
- 量子耐性がある

### 現実的な評価

Ethereum KZG Ceremonyの規模（14万人超）を考えると、「全員が結託」する現実的なリスクはほぼゼロ。Trusted Setupは理論的な弱点ではあるが、実用上の問題にはなっていない。
