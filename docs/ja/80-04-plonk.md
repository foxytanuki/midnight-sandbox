# Plonk — 証明システムの具体的な方式

## Plonkとは

**Plonk**（Permutations over Lagrange-bases for Oecumenical Noninteractive arguments of Knowledge）は2019年に発表されたZK-SNARK方式。Midnightが採用している証明システム。

主な利点:
- **Universal trusted setup**: 回路ごとのセットアップが不要。1回のセットアップで任意の回路に対応
- **効率的な検証**: 定数回のペアリング計算で完了
- **柔軟なゲート設計**: カスタムゲートで回路サイズを削減可能

## Plonk以前: Groth16との比較

| | Groth16 (2016) | Plonk (2019) |
|---|---|---|
| Setup | 回路固有（回路を変えるたびにやり直し） | Universal（1回で全回路対応） |
| 証明サイズ | 最小（3群元 = 144バイト） | やや大きい（数百バイト） |
| 検証速度 | 最速（3ペアリング） | 速い（数ペアリング） |
| 柔軟性 | 低い | 高い（カスタムゲート対応） |

Groth16は証明サイズと検証速度で勝るが、回路を変更するたびにTrusted Setupをやり直す必要がある。開発中に回路が頻繁に変わるMidnightのようなプラットフォームではPlonkが適している。

## 回路の表現

### ゲート制約

Plonkの各ゲートは以下の形式:

```
qL·a + qR·b + qO·c + qM·(a·b) + qC = 0
```

- `a, b, c`: ゲートの左入力、右入力、出力
- `qL, qR, qO, qM, qC`: **セレクター**と呼ばれる固定係数

セレクターの値を変えるだけで異なる演算を表現できる:

```
加算 (a + b = c):    qL=1,  qR=1,  qO=-1, qM=0, qC=0
乗算 (a × b = c):    qL=0,  qR=0,  qO=-1, qM=1, qC=0
定数代入 (c = 7):    qL=0,  qR=0,  qO=1,  qM=0, qC=-7
ブール制約 (a²=a):   qL=-1, qR=0,  qO=0,  qM=1, qC=0
```

### コピー制約（Permutation Argument）

ゲート間のワイヤー接続を表現する仕組み。

例: ゲート1の出力がゲート2の左入力に使われる場合、「ゲート1の c とゲート2の a は同じ値である」という制約が必要。

Plonkはこれを**置換（permutation）**で表現する:

1. 全ワイヤーに位置番号を振る
2. 「同じ値であるべきワイヤー」を巡回置換でグループ化
3. Grand Product Argument で置換の正しさを検証

これにより O(n) の追加コストで任意のワイヤー接続を表現できる。

## 証明の流れ

### Step 1: 多項式の構築

witness の値を**ラグランジュ基底**で多項式に補間する:

- `a(x)`: 全ゲートの左入力値を補間した多項式
- `b(x)`: 全ゲートの右入力値
- `c(x)`: 全ゲートの出力値

n 個のゲートがあれば、これらは次数 n-1 の多項式になる。

### Step 2: ゲート制約の多項式化

ゲート制約を多項式の等式に変換:

```
qL(x)·a(x) + qR(x)·b(x) + qO(x)·c(x) + qM(x)·a(x)·b(x) + qC(x) = 0
```

この等式が全てのゲート位置（x = ω⁰, ω¹, ..., ωⁿ⁻¹）で成り立つ必要がある。ω は n 乗根（ωⁿ = 1）。

「全てのゲート位置で 0」ということは、消失多項式 Z_H(x) = xⁿ - 1 で割り切れるということ:

```
t(x) = [ゲート制約の多項式] / Z_H(x)
```

t(x) が多項式として存在することが、全制約の充足を意味する。

### Step 3: コミットメントと評価

1. a(x), b(x), c(x), t(x) 等の多項式をKZGでコミット
2. 検証者がランダムな評価点 ζ を選ぶ（Fiat-Shamirで非対話化）
3. 証明者は各多項式の ζ での評価値と opening proof を返す
4. 検証者はペアリングで全てを一括検証

### Step 4: 検証

検証者が行う計算:
1. 公開入力から一部の値を復元
2. コミットメント、評価値、opening proof を受け取る
3. ペアリング計算で整合性を確認

検証は回路サイズに依存しない定数時間。

## カスタムゲート（Plonkish）

標準のPlonkゲートを拡張して、特定の演算を1つのゲートで表現できる:

```
標準: a + b = c（加算は1ゲート）
カスタム例: a⁵ + a = c（Poseidonハッシュのラウンドが1ゲート）
```

カスタムゲートなしだと a⁵ は乗算4回（4ゲート）必要。カスタムゲートで1ゲートに圧縮すれば回路サイズが4分の1になる。

Midnightでは暗号プリミティブ（ハッシュ、署名検証等）にカスタムゲートを使って回路を最適化している。

## Lookup Table

Plonk の拡張（plookup）で、テーブル参照を効率的に行える。

例: 8ビットのXOR演算を回路で直接計算するとビット分解が必要で数十ゲートかかる。代わりに256×256のXORテーブルを用意し「この値はテーブルに含まれている」と証明すれば1ゲートで済む。

AES暗号のS-Boxやビット演算の最適化に使われる。

## 証明サイズの内訳

典型的なPlonk証明の構成:

| 要素 | サイズ |
|---|---|
| 多項式コミットメント（a, b, c, z, t 等） | 7-9 個 × 48バイト |
| 評価値 | 5-6 個 × 32バイト |
| Opening proof | 2 個 × 48バイト |
| **合計** | **約700-900バイト** |

回路サイズが1万ゲートでも100万ゲートでも証明サイズはほぼ同じ。これが「Succinct（簡潔）」と呼ばれる理由。
