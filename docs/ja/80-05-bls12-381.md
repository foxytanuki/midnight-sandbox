# 楕円曲線 BLS12-381

[← Plonk](./80-04-plonk.md) | [次: Trusted Setup →](./80-06-trusted-setup.md)

## なぜ楕円曲線か

ZK証明（特にKZGコミットメント）は、以下の性質を持つ数学的構造を必要とする:

1. 「掛け算」は簡単だが「割り算」は困難（一方向性）
2. 値を「隠した」まま演算できる（準同型性）
3. 2つの隠された値の関係を検証できる（ペアリング）

楕円曲線暗号はこの3つを全て提供する。

## 楕円曲線の基礎

### 定義

楕円曲線は以下の形の方程式を満たす点の集合:

```
y² = x³ + ax + b  （有限体 Fp 上）
```

BLS12-381の場合: `y² = x³ + 4`（a = 0, b = 4）

### 点の加算

楕円曲線上の2点 P, Q を「加算」する演算が定義されている:

1. P と Q を結ぶ直線を引く
2. その直線が曲線と交わる第3の点を見つける
3. その点のy座標を反転した点が P + Q

この加算は:
- 結合的: (P + Q) + R = P + (Q + R)
- 可換: P + Q = Q + P
- 単位元（無限遠点 O）が存在
- 逆元が存在

→ **群**（Group）を構成する。

### スカラー倍

点 P を n 回加算する操作: nP = P + P + ... + P（n回）

**計算の非対称性（離散対数問題）**:
- nP の計算: 簡単（ダブル・アンド・アッド法で O(log n)）
- nP から n を逆算: 計算上不可能（楕円曲線離散対数問題 ECDLP）

この非対称性がKZGコミットメントの安全性の根幹。τG からτを復元できないことが、コミットメントの隠蔽性を保証する。

## BLS12-381 の構造

### なぜBLS12-381か

BLS = Barreto-Lynn-Scott。12は埋め込み次数、381はベースフィールドのビット長。

この曲線が選ばれた理由:
- **効率的なペアリング計算**: 埋め込み次数12は、セキュリティと効率のバランスが良い
- **128ビットセキュリティ**: 古典コンピュータに対して十分な安全性
- **広い採用**: Ethereum 2.0, Zcash, Filecoin, Midnightなど

### 2つの群

BLS12-381には2つの楕円曲線群がある:

| | G₁ | G₂ |
|---|---|---|
| 定義体 | Fp（素体） | Fp²（2次拡大体） |
| 点のサイズ | 48バイト（圧縮） | 96バイト（圧縮） |
| 演算速度 | 速い | 遅い（約3倍） |

ペアリングは G₁ × G₂ → GT（目標群、Fp¹² の部分群）への写像。

### パラメータ

```
p = 0x1a0111ea397fe69a4b1ba7b6434bacd764774b84f38512bf6730d2a0f6b0f6241eabfffeb153ffffb9feffffffffaaab
  ≈ 3.8 × 10¹¹⁴

群位数 r = 0x73eda753299d7d483339d80809a1d80553bda402fffe5bfeffffffff00000001
  ≈ 5.3 × 10⁷⁶
```

r はサブグループの位数で、ZK証明の演算はこの r を法とする有限体 Fr 上で行われる。

## ペアリングの詳細

### 定義

```
e: G₁ × G₂ → GT
```

双線型性:
```
e(aP, bQ) = e(P, Q)^(ab)
e(P + R, Q) = e(P, Q) · e(R, Q)
e(P, Q + S) = e(P, Q) · e(P, S)
```

### KZG検証でのペアリングの使い方

コミットメント C = f(τ)·G₁、評価値 f(a) = b、proof π = q(τ)·G₁ に対して:

検証: `e(C - b·G₁, G₂) = e(π, τ·G₂ - a·G₂)`

展開すると:
```
左辺 = e((f(τ) - b)·G₁, G₂) = e(G₁, G₂)^(f(τ) - b)
右辺 = e(q(τ)·G₁, (τ-a)·G₂) = e(G₁, G₂)^(q(τ)·(τ-a))
```

f(τ) - b = q(τ)·(τ - a) が成り立つかどうかを、τ を知らずに検証できている。

### 計算コスト

ペアリング計算は楕円曲線の点加算に比べて非常に重い:
- 点加算: マイクロ秒オーダー
- スカラー倍: ミリ秒オーダー
- ペアリング: 数ミリ秒

ただし検証で必要なペアリングは定数回（2-3回）なので、全体の検証時間は数ミリ秒で済む。

## セキュリティの仮定

BLS12-381の安全性は以下の計算困難性に依存:

1. **ECDLP（楕円曲線離散対数問題）**: nP から n を求められない
2. **co-CDH（Computational co-Diffie-Hellman）**: aG₁ と bG₂ から abG₁ を計算できない
3. **AGM（Algebraic Group Model）**: 攻撃者は群の演算を「正規の方法」でしか行えない

## 量子コンピュータの脅威

Shorのアルゴリズムにより、十分な量子ビットがあれば楕円曲線の離散対数問題は多項式時間で解ける。

影響:
- KZGコミットメントの隠蔽性が破れる
- 偽の証明が作成可能になる
- BLS署名も同様に破れる

現状の見積もり:
- BLS12-381を破るには約2,000〜4,000論理量子ビットが必要
- 2026年時点で最大の量子コンピュータは約1,000物理量子ビット（論理量子ビットは遥かに少ない）
- 実用的な脅威になるのは早くても2030年代後半と見られている

対策の方向性:
- 格子暗号ベースのコミットメント（ポスト量子暗号）への移行
- ハッシュベースの証明システム（STARK）への移行
- いずれも証明サイズ・検証時間のトレードオフがある
