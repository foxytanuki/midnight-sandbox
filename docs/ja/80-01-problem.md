## 前提: 計算の検証とは

あるプログラムに入力を与え、出力を得たとする。第三者がその出力を信用するには、普通は2つの方法しかない:

1. **再計算**: 入力とプログラムを受け取って自分で実行する
2. **信頼**: 計算した人を信用する

方法1は正確だが、入力データが見えてしまう。方法2はプライバシーは守れるが、不正の検出ができない。

ZKは第3の選択肢を提供する: **入力を見せずに、計算が正しく行われたことを数学的に証明する。**

## 何を証明するのか

ZKの文脈で「証明」とは以下を意味する:

> ある関数 `f` と公開出力 `y` に対して、「`f(x) = y` となる秘密の入力 `x` を知っている」ことを、`x` を明かさずに示す。

具体例:

| シナリオ | 関数 f | 秘密入力 x | 公開出力 y |
|---|---|---|---|
| ローン審査 | 信用スコア評価ロジック | 年収、負債、信用履歴 | 「審査通過」 |
| 年齢確認 | 生年月日 ≥ 18年前か | 生年月日 | 「18歳以上」 |
| 投票 | 有権者リストに含まれるか | 有権者ID、投票内容 | 「有効な投票」 |

## 対話型と非対話型

### 対話型ZK（Interactive ZK）

証明者と検証者がやりとりする。検証者がランダムなチャレンジを投げ、証明者が応答する。これを繰り返す。

洞窟の例え:
1. 証明者が洞窟のA側かB側からランダムに入る
2. 検証者が「A側から出てこい」と要求
3. 鍵を持っていればどちらからでも出られる
4. 20回繰り返して全部成功 → 不正の確率は 1/2^20 ≈ 0.0001%

### 非対話型ZK（NIZK: Non-Interactive ZK）

ブロックチェーンでは対話ができない（チェーン上の検証コントラクトとリアルタイムにやりとりできない）。

**Fiat-Shamir変換**で対話を排除する:
- 検証者のランダムチャレンジの代わりに、証明データ自体のハッシュを使う
- 証明者が1回の計算で証明を生成し、検証者は証明を受け取って1回の計算で検証する
- Midnightを含むほぼ全てのブロックチェーンZKシステムがこの方式

## ZKの3つの性質

1. **完全性（Completeness）**: 正しい入力を持つ証明者は、必ず検証を通過する証明を作れる
2. **健全性（Soundness）**: 正しい入力を持たない証明者は、検証を通過する証明を作れない（無視できるほど小さい確率を除く）
3. **零知識性（Zero-Knowledge）**: 検証者は「証明が正しい」以外の情報を得られない

この3つが同時に成立するのがZKの本質。どれか1つでも崩れると使い物にならない。

## ZK-SNARK と ZK-STARK

ZK証明システムの2大ファミリー:

| | ZK-SNARK | ZK-STARK |
|---|---|---|
| 正式名 | Succinct Non-interactive ARgument of Knowledge | Scalable Transparent ARgument of Knowledge |
| 証明サイズ | 小さい（数百バイト） | 大きい（数十KB〜） |
| 検証時間 | 速い | やや遅い |
| Trusted Setup | 必要（KZGベースの場合） | 不要 |
| 量子耐性 | なし（楕円曲線ベース） | あり（ハッシュベース） |
| 採用例 | Midnight, Zcash, Ethereum zkEVM | StarkNet, StarkEx |

Midnightは **Plonk（ZK-SNARKの一種）** を採用している。証明サイズの小ささと検証速度を優先した設計判断。

## Midnightにおける位置づけ

Midnightでは、開発者がCompact言語でロジックを書くと:

1. コンパイラが自動でZK回路に変換
2. ユーザーのProof Serverが証明を生成
3. チェーン上の検証コントラクトが検証

開発者はZKの数学を理解しなくてもプライバシー保護アプリを作れる。ただし「何を公開し何を隠すか」の設計判断は開発者の責任。
