## 全体像

### 1. ZKが解く問題

「計算結果は正しい」と証明したい。でも入力は見せたくない。

普通にやると「入力を渡して相手に再計算してもらう」しかない。ZKは「再計算しなくても検証できる証明データ」を作る技術。

### 2. 算術回路

プログラムのロジック（if文、比較、加算など）をすべて**多項式の制約**に変換する。

例: 「x × y = z」という計算は、そのまま1つの制約になる。「年収 ≥ 500万」は「年収 - 500万 = d（d ≥ 0）」のように分解する。

Compactで書いたコードは、コンパイラがこの変換を自動でやる。

### 3. 多項式コミットメント

証明者は制約を満たす値（witness）を持っている。これを直接見せる代わりに、**多項式に埋め込んで暗号的にコミット**する。

KZGコミットメントの直感: 多項式 f(x) の全体を1つの点（楕円曲線上の点）に圧縮する。後から「f(a) = b です」と主張したとき、コミットメントと照合して正しいか検証できる。でもコミットメントからf(x)の他の値は復元できない。

### 4. Plonk

Midnightが使っている証明システム。

- **Universal trusted setup**: 回路ごとにセットアップし直す必要がない
- **ゲート構造**: 各ゲートは `aL × aR = aO` の形で表現
- **コピー制約**: ワイヤー間の接続を置換（permutation）で表現

### 5. 楕円曲線 BLS12-381

KZGコミットメントの土台。ペアリング対応の楕円曲線で、128ビットセキュリティを提供する。Ethereum 2.0のBLSシグネチャでも同じ曲線を使っている。量子耐性はない。

### 6. Trusted Setup

KZGには初期パラメータ（SRS）が必要。これを生成する過程で使った乱数を誰かが知っていると偽の証明が作れてしまう。対策としてMPCセレモニーで数百〜数千人が順番に乱数を混ぜる。

### 7. 証明ライフサイクル

```
Compact コード
  ↓ コンパイル
算術回路（制約システム）
  ↓ + witness（秘密の入力値）
Proof Server（ローカル）が証明生成
  ↓
証明データ（数百バイト〜数KB）
  ↓ トランザクションに添付
オンチェーンの検証コントラクトが検証
  ↓ ペアリング計算で O(1) 検証
OK / NG
```

証明生成は重い（秒〜十数秒）が、検証は軽い（ミリ秒）。この非対称性がZKの実用性を支えている。
