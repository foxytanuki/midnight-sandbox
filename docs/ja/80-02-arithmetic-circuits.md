# 算術回路 — 計算を数学に変換する

[← ZKが解く問題](./80-01-problem.md) | [次: 多項式コミットメント →](./80-03-polynomial-commitment.md)

## なぜ回路が必要なのか

ZK証明システムは任意のプログラムをそのまま扱えない。プログラムを**有限体上の算術演算**（加算と乗算）の組み合わせに変換する必要がある。この変換後の表現を**算術回路（Arithmetic Circuit）**と呼ぶ。

電気回路と同じ発想:
- 入力ワイヤーから値が入る
- ゲート（加算ゲート、乗算ゲート）で演算する
- 出力ワイヤーに結果が出る

## 有限体（Finite Field）

算術回路の演算は**有限体 Fp** 上で行われる。

有限体とは:
- 0 から p-1 までの整数の集合（pは大きな素数）
- 加算と乗算が定義されていて、結果は常に mod p で 0〜p-1 に収まる
- BLS12-381 の場合、p は約 2^255 の素数

なぜ有限体か:
- 整数のまま計算すると値が際限なく大きくなる
- 有限体なら全ての値が固定サイズに収まる
- 暗号学的な性質（離散対数問題の困難さなど）が利用できる

## 基本的な変換の例

### 加算

```
z = x + y
```

そのまま1つの加算ゲートになる。

### 乗算

```
z = x * y
```

そのまま1つの乗算ゲートになる。

### 比較（x ≥ 500万）

比較は直接表現できない。ビット分解して処理する:

```
d = x - 5000000        // d ≥ 0 であればよい
b[0], b[1], ..., b[n] = bits(d)   // dをビット分解
各 b[i] × (1 - b[i]) = 0 を制約   // 各ビットが0か1であること
b[0] + 2*b[1] + 4*b[2] + ... = d  // ビットの合計がdと一致
```

これで「d が非負整数である」ことが制約として表現される。比較1つで数十〜数百の制約が生まれる。

### 条件分岐（if-else）

```
if (condition) { z = a } else { z = b }
```

ZK回路に分岐はない。**両方を計算して選択する**:

```
z = condition * a + (1 - condition) * b
```

`condition` は 0 か 1 に制約される。

## R1CS（Rank-1 Constraint System）

算術回路を形式化した表現。各制約は以下の形:

```
(a₁x₁ + a₂x₂ + ...) × (b₁x₁ + b₂x₂ + ...) = (c₁x₁ + c₂x₂ + ...)
```

つまり「左辺の線形結合 × 右辺の線形結合 = 出力の線形結合」。

これを行列で表すと:

```
L · w ⊙ R · w = O · w
```

- `w`: witness ベクトル（全ての変数の値）
- `L, R, O`: 制約を定義する行列
- `⊙`: 要素ごとの乗算

R1CSはGroth16などの古い証明システムで使われる。Plonkは別のゲート表現を使う（後述）。

## Plonkのゲート表現

Plonkでは各ゲートを以下の形で表す:

```
qL·aL + qR·aR + qO·aO + qM·(aL·aR) + qC = 0
```

- `aL, aR, aO`: 左入力、右入力、出力の値
- `qL, qR, qO, qM, qC`: セレクター（どの種類のゲートかを決める係数）

例:

| 演算 | qL | qR | qO | qM | qC |
|---|---|---|---|---|---|
| 加算: aL + aR = aO | 1 | 1 | -1 | 0 | 0 |
| 乗算: aL × aR = aO | 0 | 0 | -1 | 1 | 0 |
| 定数: aO = 42 | 0 | 0 | 1 | 0 | -42 |

R1CSより柔軟で、カスタムゲートの追加も容易。

## Witness（証人）

回路の全ワイヤーに具体的な値を割り当てたものを **witness** と呼ぶ。

- 公開入力（public input）: 検証者も知っている値
- 秘密入力（private input / secret witness）: 証明者だけが知っている値
- 中間値: 計算過程で生じる値

証明者は「全ての制約を同時に満たすwitness を知っている」ことを証明する。witness自体は証明に含まれない。

## 回路サイズとパフォーマンス

| 指標 | 意味 |
|---|---|
| ゲート数 | 回路内の制約の数。多いほど証明生成が遅い |
| ワイヤー数 | 変数の数 |
| 証明時間 | ゲート数にほぼ比例（O(n log n)） |
| 検証時間 | ゲート数にほぼ依存しない（O(1)） |

典型的な規模:
- 単純なハッシュ検証: 数千ゲート
- 署名検証: 数万ゲート
- ローン審査のようなビジネスロジック: 数万〜数十万ゲート

## Compactとの関係

Compact言語で書いたコードは、コンパイラが以下の変換を行う:

```
Compact ソースコード
  ↓ パース・型チェック
中間表現（IR）
  ↓ 最適化
Plonk ゲート制約
  ↓
証明キー + 検証キー
```

開発者は回路を直接書かない。ただし、回路サイズに影響する書き方（ループ回数、条件分岐の深さなど）は意識する必要がある。ループは展開されるので、`for (i = 0; i < 1000)` は1000倍の制約になる。
