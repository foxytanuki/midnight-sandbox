# ZK Loan で理解する「なぜ ZK が必要か」

> **対象読者**: スマートコントラクト開発経験があり、ZK の必要性を概念的に理解したい開発者。暗号学の深い知識は不要。
>
> **参考リポジトリ**: [midnightntwrk/example-zkloan](https://github.com/midnightntwrk/example-zkloan) — 本ドキュメントで解説するコード例はすべてこのリポジトリに基づく。

---

## 前提: ローン審査のプライバシー問題

ローンを借りるには信用スコア・収入・勤続年数を貸し手に開示する必要がある。ブロックチェーン上でこれをやると、公開台帳に全員分の個人情報が記録され、誰でも閲覧できてしまう。

**ZK のない世界の選択肢：**

- オンチェーンに生データを載せる → プライバシーなし
- オフチェーンで審査してオンチェーンには結果だけ載せる → 審査がブラックボックス

ZK はこの二択を崩し、 **データを隠したまま、審査の正しさを証明する**ことを可能にする。

## アクター構成と信頼モデル

### 登場人物

```
┌─────────────┐     署名依頼      ┌──────────────────┐
│   借り手     │ ──────────────→  │  データプロバイダ  │
│ (Applicant) │ ←────────────── │  (Attestation)    │
│             │   署名付きデータ   │                  │
└──────┬──────┘                  └──────────────────┘
       │                            ↑ 登録/削除
       │ ローン申請                  │
       │ (ZK証明付き)               │
       ▼                           │
┌──────────────────────────────────┴───┐
│         コントラクト (on-chain)        │
│  - 審査ロジック（Tier判定）固定         │
│  - プロバイダ管理                      │
│  - ローン記録                         │
└──────────────────┬───────────────────┘
                   │ デプロイ・管理
                   ▼
             ┌───────────┐
             │  貸し手     │
             │  (Admin)   │
             └───────────┘
```

### 各アクターの役割・知れる情報・知れない情報

| アクター | 役割 | 知れること | 知れないこと |
|---|---|---|---|
| **貸し手（Admin）** | コントラクトのデプロイ・管理、信頼するプロバイダの登録 | 承認/却下、ローン金額 | 信用スコア、収入、勤続年数 |
| **データプロバイダ** | 信用情報の発行元として、データの真正性を署名で担保 | 借り手の生データ（スコア・収入等） | ローン申請の有無・時期・金額 |
| **借り手** | プロバイダから署名を取得し、ローンを申請 | 自分のデータすべて | — |

**従来の金融との違い**: 従来は信用調査機関だけでなく貸し手にも審査情報が渡る。この設計では「プロバイダには情報を渡すが、貸し手には結果以外知られたくない」が成立する。

## インフラ構成: 2 つのサーバーの区別

このシステムには **運用者が異なる 2 つのサーバー**が登場する。混同しやすいので明確に区別する。

### Attestation API（プロバイダが運用）

信用情報プロバイダ（信用調査機関等）が運用する API サーバー。

```
借り手 → POST /attest → プロバイダが自社DBからスコアを取得・検証 → 署名を返す
```

- プロバイダは信用データの **発行元**であり、借り手の生データを見ることは前提
- 署名は「このデータは正しい」というプロバイダのお墨付き
- **ローン申請の前に一度だけ** 行う事前準備

> **注**: サンプルコードの `POST /attest` は借り手がデータを送信する形になっているが、本番ではプロバイダが自社 DB からデータを取得して署名する想定。サンプルの簡略化によるもの。

### Proof Server（借り手が運用）

借り手のマシン上で動くローカルサーバー（`http://localhost:6300`）。ZK 証明の生成に特化した計算エンジン。

```
dApp → [署名付き信用データ + ウォレット鍵 + PIN + 申請額] → Proof Server → ZK証明
```

借り手側で動かす理由は、証明生成に **借り手の秘密情報**が必要だから：

| 証明に必要な入力 | 持ち主 | 外部に渡せるか |
|---|---|---|
| 署名付き信用データ | プロバイダが発行 → 借り手が保持 | 渡すとプライバシーが崩れる |
| ウォレット秘密鍵 | 借り手 | 渡すと資産を盗まれる |
| 秘密 PIN | 借り手 | 渡すと本人性が崩れる |
| 申請額 | 借り手 | 渡すとローン申請を知られる |

### フロー全体

```
【事前準備】                        【ローン申請時】
プロバイダの Attestation API        借り手のローカル Proof Server

  借り手 → プロバイダ                 借り手のマシン上で:
  「署名をください」                    署名付き信用データ
        ↓                            + ウォレット鍵 + PIN + 申請額
  プロバイダ: 自社DBで検証                    ↓
  → 署名を返す                        Proof Server: ZK証明を生成
                                            ↓
  (プロバイダの仕事はここで終わり)      オンチェーンに ZK証明を送信
                                            ↓
                                      Node: 証明を検証 → 結果を記録
                                      (貸し手は結果だけ見える)
```

## ZK が解決する 3 つのこと

### 1. プライバシー保護

信用スコア・収入等の生データは ZK 回路内（witness）でのみ使われ、オンチェーンには一切出ない。

```
オンチェーンに公開される情報:
  - ローンステータス: Approved / Rejected / Proposed
  - 承認額: 10000

オンチェーンに出ない情報:
  - 信用スコア: 720
  - 月収: $2,500
  - 勤続月数: 24
  - PIN
```

ただし、承認額からティアが推測できるというトレードオフはある（後述）。

### 2. 評価ロジックの透明性

審査基準（ティア判定）はコントラクトに固定されている（`zkloan-credit-scorer.compact` L103-118）。

```
Tier 1: スコア ≥ 700 かつ 月収 ≥ $2,000 かつ 勤続 ≥ 24ヶ月 → 最大 $10,000
Tier 2: スコア ≥ 600 かつ 月収 ≥ $1,500                    → 最大 $7,000
Tier 3: スコア ≥ 580                                       → 最大 $3,000
それ以外 → Rejected
```

もしプロバイダが直接 approve/reject を返す設計だと、この判定がプロバイダのブラックボックスになる。ZK を使うことで、 **生データの真正性はプロバイダが担保し、判定ロジックはコントラクトが公開・固定する**という責務分離が成立する。

### 3. 計算の正しさの暗号学的保証

ZK 証明により「コントラクトに書かれたロジックが改ざんなく正しく実行された」ことが保証される。

借り手がローカルで証明を生成するが、データの改ざんはできない。ZK 回路内でプロバイダの Schnorr 署名を検証しており（`compact` L101）、プロバイダが署名したデータと異なる入力では有効な証明が生成できない。

```
借り手が嘘のスコアで証明を作ろうとした場合:
  入力: creditScore=800, 署名=プロバイダが creditScore=600 に対して発行
  → 回路内で署名検証 → メッセージと署名が不一致 → 証明生成不可

借り手が署名を偽造しようとした場合:
  → プロバイダの秘密鍵を持っていないので Schnorr 署名を偽造できない
```

## ZK が解決しないこと

### データの真正性はプロバイダへの信頼

「信用スコアが本当に 720 か」は ZK では担保できない。プロバイダが嘘のスコアを署名すれば、コントラクト側では検知できない。

```
ZK が担保する範囲:
  ✓ 署名がプロバイダ本人のものか（Schnorr署名検証）
  ✓ データが改ざんされていないか（署名に紐づく）
  ✓ ティア判定が正しく実行されたか（ZK回路）
  ✗ スコアが本当に720か → プロバイダを信頼するしかない
```

これは従来の金融と同じ構造。銀行は信用調査機関のスコアを信頼して使っている。このシステムの違いは、信頼の先の **プライバシー**と**評価の透明性**を暗号学的に保証する点。

緩和策として、複数プロバイダの登録が可能で、admin が不正プロバイダを排除できる設計になっている。

### 承認額からの情報漏洩

承認額とティア判定ロジックは公開されているため、承認額から借り手の属性の **範囲**が推測できる。

```
承認額 $10,000 → Tier 1 → スコア≥700, 月収≥$2,000, 勤続≥24ヶ月
承認額 $7,000  → Tier 2 → スコア≥600, 月収≥$1,500
承認額 $3,000  → Tier 3 → スコア≥580
```

ただし正確な値はわからない（スコアが 700 なのか 800 なのかは不明）。また、申請額 ≤ 承認上限の場合は申請額がそのまま承認額になるため、ティアの特定すらできないケースもある（$5,000 承認 → Tier 1 か Tier 2 か不明）。

これは `disclose` で何を公開するかというコントラクト設計上のトレードオフ。より強いプライバシーが必要なら、承認額を公開せず approve/reject のみ返す設計も可能。

## このシステムが意味を持つ条件

ZK Loan の 3 つの価値（プライバシー・透明性・計算の正しさ）は独立しており、すべてが常に必要とは限らない。

| ケース | プライバシー | 透明性 | 計算の正しさ |
|---|---|---|---|
| **すべてのアクターが別主体** | 有効 | 有効 | 有効 |
| **プロバイダと貸し手が同一** | 薄れる（貸し手は既にデータを持つ） | **依然有効**（借り手に公平な審査を保証） | 有効 |
| **信頼できるプロバイダが不在** | 有効 | 有効 | 有効だが **入力が嘘なら無意味** |
| **審査基準が動的・主観的** | 有効 | **不可**（ZK回路に固定できない） | 有効 |

**最も効果を発揮する前提:**
1. 信頼できるデータ発行者がいる
2. 評価基準が機械的に固定できる
3. データ発行者と貸し手が別主体（プライバシーの価値が最大化）

## コントラクトの審査ロジック

実際のティア判定コード（`contract/src/zkloan-credit-scorer.compact` L85-119）：

```compact
circuit evaluateApplicant(userPubKeyHash: Field): [Uint<16>, LoanStatus] {
    // witness から借り手の信用情報・署名・プロバイダIDを取得（ZK回路内、オンチェーンに出ない）
    const [profile, signature, providerId] = getAttestedScoringWitness();

    // プロバイダが admin に登録済みか確認
    assert(providers.member(disclose(providerId)), "Attestation provider not registered");
    const providerPk = providers.lookup(disclose(providerId));

    // 署名検証用メッセージ: [creditScore, monthlyIncome, monthsAsCustomer, userPubKeyHash]
    const msg: Vector<4, Field> = [
        profile.creditScore as Field,
        profile.monthlyIncome as Field,
        profile.monthsAsCustomer as Field,
        userPubKeyHash
    ];

    // プロバイダの Schnorr 署��を検証 — データの改ざん防止
    Schnorr_schnorrVerify<4>(msg, signature, providerPk);

    // Tier 1: Best applicants
    if (profile.creditScore >= 700 && profile.monthlyIncome >= 2000 && profile.monthsAsCustomer >= 24) {
        return [10000, LoanStatus.Approved];
    }
    // Tier 2: Good applicants
    else if (profile.creditScore >= 600 && profile.monthlyIncome >= 1500) {
        return [7000, LoanStatus.Approved];
    }
    // Tier 3: Basic eligibility
    else if (profile.creditScore >= 580) {
        return [3000, LoanStatus.Approved];
    }
    else {
        return [0, LoanStatus.Rejected];
    }
}
```

処理の流れ：

1. `witness` から信用データを取得（オフチェーン、秘密）
2. プロバイダの登録確認
3. Schnorr 署名検証（データの真正性を ZK 回路内で暗号学的に確認）
4. ティア判定（生データを使って判定するが、`disclose` しないのでオンチェーンに出ない）

呼び出し元の `requestLoan`（L49-62）で、返り値の承認額とステータスだけが `disclose` されて公開される。

## コントラクトの管理構造

```compact
constructor() {
    admin = ownPublicKey();  // デプロイ者が初期 admin
}
```

admin（= 貸し手）ができること：

| 操作 | circuit |
|---|---|
| プロバイダの登録 | `registerProvider(providerId, providerPk)` |
| プロバイダの削除 | `removeProvider(providerId)` |
| ユーザーのブラックリスト追加 | `blacklistUser(account)` |
| ブラックリスト解除 | `removeBlacklistUser(account)` |
| admin 権限の移譲 | `transferAdmin(newAdmin)` |

貸し手が「どのプロバイダを信頼するか」を決め、審査基準はコントラクトに固定される。

## まとめ

```
従来のオンチェーンローン:
  借り手 → [生データ公開] → コントラクト → 貸し手が全部見える

ZK Loan:
  借り手 → [プロバイダから署名取得] → [ローカルで ZK 証明生成] → コントラクト → 貸し手は結果だけ見える
```

| 観点 | 従来 | ZK Loan |
|---|---|---|
| 貸し手に見える情報 | 全部 | 承認結果と金額のみ |
| 審査ロジック | 貸し手 or プロバイダのブラックボックス | コントラクトで公開・固定 |
| 計算の正しさ | 信頼ベース | 暗号学的に証明 |
| データの正しさ | プロバイダを信頼 | プロバイダを信頼（同じ） |
| データの改ざん防止 | なし | 署名検証が ZK 回路に組み込み |

ZK は万能ではない。だが「プライバシー」「透明性」「計算の正しさ」を同時に実現する唯一の手段であり、従来の「透明性のためには情報を公開するしかない」という制約を根本から変える。
