# Midnight 冷静な評価

Midnightの価値提案を、肯定も否定もせず検証した記録。

---

## Midnightの本質

Midnightが提供するのは「プライバシー」ではなく、より正確には以下の実行モデルである。

> 生データを相手に渡さずに、共有ルールに従って正しく判定したことを、複数主体が同じ台帳上で検証できる

つまり「データ共有」ではなく「判定共有」の基盤。

従来の業務では二択しかなかった。

1. 生データを相手に渡して透明性を取る
2. 相手を信じてブラックボックス審査にする

Midnightは第三の選択肢を作る。

- データの真正性 → データプロバイダの署名で担保
- 判定ロジック → コントラクトに固定
- 正しく実行されたこと → ZK証明で保証

この責務分離がMidnightの設計上の核。

---

## 本当の差別化ポイント

### 1. ZKアプリの開発枠組みが一体化している

Ethereum上でもZKは使えるが、Circom/Noir + verifier + private state管理 + wallet連携を全部自前で持つ必要がある。MidnightはZK回路生成・証明・秘密状態管理・shielded tokenまで含めた開発枠組みを最初から持っている。

Midnightの本質的な価値はクロスチェーンではなく、この開発体験にある。

### 2. 選択的開示

完全匿名ではなく「何を公開し、何を隠すか選べる」設計。企業が必要とするのはたいてい「全部隠す」ではなく「必要な人には見せたい、でも全世界には見せたくない」であり、ここは思想として企業寄り。

### 3. ローカル実行モデル

秘密入力はユーザーの端末で保持し、証明だけをオンチェーンに送る。秘密情報が外部に出ない設計であり、Proof Serverがローカルで動く。

## 弱点

### 1. 必要条件が厳しい

Midnightが意味を持つには以下が同時に必要。

- 複数社が関与する
- 互いに全部は信用していない
- 生データは出せない
- でも判定結果の共有状態は必要
- 中央管理者に全部見せると事業が壊れる

この条件が揃う業務は現実にはかなり狭い。

### 2. 代替手段で足りることが多い

多くの案件は以下で解決できる。

- **DB + 権限管理 + 監査ログ**: 単一企業の業務はほぼこれで足りる
- **API連携 + 電子署名**: 企業間データ受け渡しはこれで回る
- **Verifiable Credentials (VC)**: 年齢確認、資格証明などの静的な証明
- **Confidential Computing / TEE**: データを見せずに計算したいだけなら
- **Ethereum上のZK**: 技術的にはEthereum単体でも実現可能

### 3. ZKが保証しないもの

ZKは「正しい計算」を保証するが「正しい入力」は保証しない。データプロバイダが嘘のデータに署名した場合、ZKでは防げない。これは従来の金融における信用調査機関への信頼と同じ構造。

### 4. 証明生成の重さ

クライアント側で数秒〜数十秒。モバイル対応は課題。リアルタイム性や軽快なUXが求められる用途には向かない。

### 5. プライバシーも万能ではない

- NIGHTトークンの取引は公開される（Shielded操作のみ秘匿）
- Indexerやviewing keyの運用次第で見える範囲が変わる
- 承認結果から属性が推測される場合がある（設計判断次第で緩和可能）

### 6. エコシステムの規模

Ethereumと比較して小さい。ツール成熟度、ドキュメント、コミュニティは発展中。

### 7. クロスチェーンの現状

- 現時点ではCardanoとの接続が中心
- Ethereum/Solana等との接続はロードマップ段階
- ブリッジを経由するとZKの暗号学的保証がブリッジへの信頼に置き換わる

---

## Midnightが不要なケース

以下に該当する場合、Midnightは基本的に不要。

- 参加者が実質1社で、中央管理に問題がない
- 必要なのは「証明」だけで「更新され続ける共有状態」ではない
- 秘匿計算は必要だが共有台帳は不要（TEE/MPCで足りる）
- 低コスト・低遅延・運用の単純さが最優先
- メタデータ漏洩で事業が致命傷を受けるほどではない
- 「ブロックチェーン」が目的になっている

---

## Midnightが検討に値するケース

以下の条件が複数揃う場合に限り、検討対象になり得る。

- 複数主体が関与し、単一運営者を置きたくない
- 生データは出せないが、判定結果の正しさは検証させたい
- 監査人・規制当局・取引先にブラックボックスではない説明が必要
- 共有状態をオンチェーンで継続的に運用したい
- 信頼できるデータ発行者が存在する
- 評価基準が機械的に固定できる

### 具体的な業務領域

- 競合企業間の不正検知・制裁リスク共有
- 秘匿入札・RFQ・相対取引市場
- プライベート給付・保険適用の二重利用防止
- サプライチェーン金融・二重担保防止
- 取引先適格性・入札資格の証明
- 規制適合判定の共有

ただしこれらも、TEE/VC/既存DBで解決可能な場合がある。Midnight固有に必要になるのは、「秘密のまま複数主体でstatefulな状態を共有し続ける」必要がある場合のみ。

---

## 「パッとしない」理由

Midnightの価値提案が弱く感じられるのは、技術が悪いからではなく、以下の構造的な理由による。

1. **痛みがまだ深くない**: 多くの企業はDB + API + 監査ログで回っている。「今のやり方だと事業が止まる」ほど困っている会社が少ない
2. **必要条件が厳しすぎる**: 5つの条件が同時に揃う案件は現実に少ない
3. **導入コストとのバランス**: 仮に課題があっても、既存システムの改善で逃げられることが多い
4. **制度・慣行が追いついていない**: 技術は先にあるが、需要と導入条件がまだ揃い切っていない

---

## 総合評価

Midnightは技術的に筋の通った設計をしている。「判定共有」というコンセプトと、ZKアプリの開発枠組みの一体化は嘘のない差別化である。

しかし、現時点で「Midnightがないと成立しない」業務はほぼ存在しない。多くの案件は代替手段で足りる。価値が出るのは「秘密のまま複数主体でstatefulな共有が必要」という狭い条件下に限られる。

万能な基盤ではなく、特定条件下で意味を持つ専用ツールとして評価するのが妥当。
