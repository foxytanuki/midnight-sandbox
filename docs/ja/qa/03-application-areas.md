# Midnight 応用領域の探索

02-critical-assessment.md の評価を前提に、Midnightの性質が実際に効く可能性のある領域を調査した記録。空想ではなく、既存の業界課題から逆算して特定する。

---

## 前提条件

Midnightが意味を持つのは以下の条件が複数揃う場合に限られる（02-critical-assessment.md より）。

- 複数主体が関与し、単一運営者を置きたくない
- 生データは出せないが、判定結果の正しさは検証させたい
- 更新され続ける共有状態が必要
- 評価基準が機械的に固定できる
- 信頼できるデータ発行者が存在する

以下の各領域について、この条件に照らして評価する。

---

## 領域1: サプライチェーン金融 / 二重担保防止

### 現実度: 高

### 今どうやっているか

銀行・ファクター・SCF事業者が個別に審査している。ERP、電子請求書、船荷証券（B/L）、倉庫証券、債権譲渡登記、取引先確認、監査で確認する。一部の国・業界ではレジストリやEDIでカバーされるが、金融機関横断でリアルタイムに重複を防ぐ仕組みは弱いことが多い。

### 何が痛いか

- 同じ請求書・在庫・船荷証券が複数の金融機関に差し入れられる（二重担保・二重譲渡）
- 各金融機関は競争上、取引先名・条件・回転率・価格情報を出したくない
- クロスボーダーだとレジストリが分断される
- 不正が起きると損失が大きいのに、共有したくない情報が多すぎる
- 手作業照合・監査コストが高い

### Midnightの性質がどう効くか

MidnightのCommitment/Nullifierモデルと相性が良い。

- データ発行者: ERP、e-invoice基盤、倉庫/物流、売掛先承認基盤
- 判定ルール: 実在する債権か、金額範囲は上限内か、期限内か、既に担保化/譲渡済みでないか
- 共有状態: 「この債権は既に使われた」という状態だけを共有

生の請求書や取引先名を出さずに、「有効で未使用の債権である」ことだけを複数金融機関が共有できる。これは「判定共有」であり、かつstatefulである。

### 代替手段の評価

- **中央DB/業界レジストリ**: 十分な国・市場ではこれで終わる。ただし中央運営者に競争情報が集まる問題と、市場横断/国際横断でカバレッジが弱い問題がある
- **VC**: 「この請求書は存在する」は証明できるが、二重利用防止の共有状態管理は弱い
- **TEE**: 技術的にはかなり有力な代替。ただし「誰のTEEか」「運営者をどこまで信じるか」が残る
- **API連携**: 双務連携なら可能。多者間・競合間だとスケールしにくい

### 実現可能性

高め。今のMidnightでPoCは十分可能。

- ルールが固定的で機械化しやすい
- B2Bなので証明生成の数秒は許容される
- モバイル不要
- クロスチェーン不要
- 共有状態が本質

限界: 法的権利移転そのものは別問題。データ発行者の不正入力は防げない。係争、訂正、償還、回収などの業務例外はオフチェーン設計が必要。

## 領域2: 金融機関間の不正・制裁リスク共有

### 現実度: 中

### 今どうやっているか

各社のAML/不正検知システム、制裁スクリーニングベンダー、共同ブラックリスト、当局への届出、一部の情報共有セーフハーバー制度で対応している。

### 何が痛いか

- 本当は他社と共有したいが、顧客データ・取引データを共有できない
- 銀行秘密、個人情報規制、訴訟リスクが障壁
- 共有が遅い、粒度が粗い、説明責任が弱い
- 一行で弾かれた主体が別行へ移る（whack-a-mole問題）

### Midnightの性質がどう効くか

高度なAML解析全体ではなく、狭い判定共有に効く。

- 「この主体は共有ルール上high-riskに該当した」
- 「この口座/IDは既に別機関で不正利用フラグ済み」
- 「k-of-nの独立機関がリスク判定した」
- 「この送金は共有された制裁ルールに抵触した」

生の取引グラフは出さず、ルール適合結果だけ共有する使い方。

### 代替手段の評価

- **中央ユーティリティ/共同DB**: 一番現実的。ただしデータ集中リスクと競争・責任問題がある
- **TEE / confidential computing**: かなり強い代替。単一運営者を置けるならこちらが勝つことが多い
- **MPC / federated analytics**: 高度分析ではこちらが自然
- **VC**: 動的リスク状態には弱い

### 実現可能性

中程度。狭いルール共有なら可能。AML基盤全体は無理。

弱点: 不正判定はしばしば主観的・動的。誤検知時の異議申立てや証拠開示が必要。ガバナンスが技術より難しい。大手銀行は結局「契約できる中立運営者」を選びがち。

## 領域3: 秘匿RFQ・相対取引・企業調達入札

### 現実度: 中

### 今どうやっているか

調達SaaS、電話・メール・ブローカー、プライベートRFQプラットフォーム、取引所/オークション運営者を信頼する形で運営されている。

### 何が痛いか

- 入札価格、数量、意図、顧客名が運営者に見える
- 先回り・価格操作・談合疑念がある
- 「本当にルール通り選定したのか」がブラックボックス
- 相対市場では板を公開したくない

### Midnightの性質がどう効くか

- 入札内容を秘匿したまま、入札資格・証拠金・上限遵守だけを証明
- ルールに従って勝者選定したことを共有
- 落札/約定状態だけを共有
- 入札提出、締切、落札、取消、執行、失効といった継続的な状態管理

### 代替手段の評価

- **中央プラットフォーム**: 多くの場合これで十分。運営者を信頼できるなら問題ない
- **TEE**: かなり有力。単一の市場運営者が受け入れられるなら優勢
- **既存の秘密入札方式**: ユースケースによっては十分

### 実現可能性

中〜高。PoCはやりやすい。ただし問題は技術より市場。既にプラットフォームを信頼している市場では不要。本当に痛いのは「情報漏洩への不信があるが共通オペレータを置きたくない市場」だけ。

B2B調達やコモディティ相対市場の方が、DeFi系より現実的。

## 領域4: 保険給付・補助金・医療給付の二重利用防止

### 現実度: 低〜中

### 今どうやっているか

保険請求clearinghouse、支払基金/審査支払機関、個別保険者DB、不正検知ベンダー、手作業審査。

### 何が痛いか

- 二重請求、給付上限超過、本人なりすまし
- PHI/医療情報を横断共有しにくい
- 保険者間調整コストが高い

### Midnightの性質がどう効くか

効くのは狭いケース。

- 「この人はこの給付条件を満たす」
- 「この給付枠は未使用/残枠あり」
- 「この請求IDは未使用」
- 診断名や全医療情報は秘匿

給付資格と消費状態の共有に限定される。

### 代替手段の評価

- **中央clearinghouse**: 現実世界では強い
- **TEE**: 高度照合にはかなり自然
- **VC**: 資格証明には使えるが利用回数管理が弱い

### 実現可能性

PoCはできるが本番化が重い。ルール例外が多い。異議申立て・再審査が多い。既存clearinghouseが強い。監査・法令対応で結局詳細開示が必要になることが多い。

やるなら福利厚生クーポン、限定補助金、給付バウチャーのような狭い制度から。

## 領域5: 取引先適格性・サプライヤー資格パスポート

### 現実度: 低

### 今どうやっているか

監査証跡提出、サプライヤーポータル、第三者認証、ベンダーマスタ、電子署名付き証明書。

### 何が痛いか

- 同じ監査資料を何社にも提出する手間
- 機密文書を見せたくない
- 更新・失効管理が面倒

### Midnightの性質がどう効くか

「資格条件を満たす」ことだけを共有し、閾値判定だけを公開できる。

### 代替手段の評価

- **VCが強い**: 静的資格証明にはほぼ十分
- **DB/API**: 元締めがいるなら十分

### 実現可能性

技術的には高いが、商業的には弱い。多くの場合VCで終わる。

Midnightが必要になるのは、資格が常時変化し、複数主体が共有状態を持ち、回数制限や利用状態、取消状態を同期したいときだけ。「資格証明」だけなら弱い。「資格 + 状態共有」があるときだけ候補。

## 代替手段との境界線

各代替手段が勝つ条件を整理する。

### DB + APIが勝つ条件

- 実質1社運営
- 参加者が運営者を受け入れる
- 監査ログがあれば十分
- 高速性・運用容易性が重要

### VCが勝つ条件

- 1回限り or 静的属性証明
- 更新され続ける共有状態が不要
- 年齢確認、資格、会員証明など

### TEE / confidential computingが勝つ条件

- 重い計算が必要
- 動的ルール
- 1つの中立運営者を置ける
- 生データは見せたくないが共有台帳は不要

### Ethereum等で直接ZKが勝つ条件

- 既にそのチェーンの流動性/エコシステムが必要
- ZK実装力がある組織
- Midnight外との接続が主目的

### Midnightが残る条件

- 上記のいずれでも不十分
- 秘密のまま複数主体でstatefulな状態を共有し続ける必要がある
- 中央運営者を置きたくない or 置けない

## 業界トレンドとの接点

### 追い風

1. **データ主権・越境規制・機密保持圧力**: 企業は「データを共有したい」のではなく「共有せずに協調したい」場面が増えている。特に金融・ヘルスケア・B2B取引
2. **企業間DXの進展**: ERP、e-invoice、電子B/L、サプライヤー審査、制裁スクリーニング等のデジタル化が進むと、判定ルールを機械化しやすくなる。Midnightの前提条件が揃いやすくなる
3. **二重利用・重複計上への規制圧力**: 金融犯罪、補助金不正、二重ファイナンス、給付の重複請求。ここは本当に痛みがある
4. **AI時代のガバナンス判定**: AI自体ではなく「このデータはこの目的で使ってよいか」「このモデルはこの制約を満たすか」の判定需要。ただし多くの場合TEE/ポリシー基盤が先に勝つ

### 向かい風

1. **中立SaaS/業界ユーティリティの方が導入しやすい**: 多くの企業は暗号学的中立性より、契約できる運営会社を選ぶ
2. **Confidential cloud / TEE / clean roomの台頭**: Azure/AWS/GCPのconfidential computingやデータクリーンルームが伸びている。Midnightの競争相手はブロックチェーンではなくこちら
3. **Mandatory registry / e-invoicingの進展**: 国や業界で中央レジストリが整うほど、Midnightの必要性は下がる

## タイミングの評価

### 全体としては「まだ早い」

- 需要が広く顕在化していない
- 企業の第一選択はまだDB/API/TEE
- ガバナンス問題の方が技術問題より大きい
- Midnight自体のエコシステムが小さい
- クロスチェーンは未成熟
- 証明生成UXはまだ重い

### 既に他手段が先行している領域

- 単純な資格証明 → VC
- KYCパスポート → VC/既存ベンダー
- Ethereum流動性前提のDeFiプライバシー → Aztec等
- AIデータ市場 → 市場自体が成立していない

### 今からPoCに値する領域

- 二重担保防止（サプライチェーン金融・貿易金融）
- 競合間の限定的リスク共有
- ルール固定のB2Bワークフロー

一般論としては早い。だが「二重利用防止 × 多者間 × 機密データ」だけは今でもリアルな痛みがある。

## 最も現実的な最初の一歩

もしMidnightで1つだけPoCをやるなら、**売掛債権ファイナンスの二重担保防止ネットワーク**が最も説得力がある。

### なぜこれか

- 痛みが具体的で金銭損失が明確
- 多者間の業務である
- 競争上データを出したくない構造が本質
- 共有状態（encumbered / available）が核
- ルール化しやすい
- B2BなのでUX遅延を許容しやすい
- MidnightのCommitment/Nullifier設計と相性が良い
- クロスチェーン不要で今の技術で完結する

### PoCで証明すること

- ある請求書/売掛債権が実在すること
- 一定条件を満たすこと
- まだ担保利用されていないこと
- 利用上限内であること
- 請求書本文、取引先名、価格条件は秘匿

### 共有する状態

- encumbered / available
- 使用済みnullifier
- 担保上限の消費状態
- 取消/失効状態

### 限界

- 不正入力は防げない（データ発行者の信頼に依存）
- 法的執行は別途必要
- 例外処理はオフチェーン設計が必要
- 「台帳」ではなく「共同利用防止レイヤー」として割り切るべき

---

## 総括

Midnightの応用領域は狭い。だが、その狭い場所には本当の痛みがある。

最も現実的なのはサプライチェーン金融の二重担保防止であり、これは技術適合度、市場の痛み、PoC実現可能性のすべてが揃う。それ以外の領域は、代替手段で足りるか、導入障壁が高すぎるか、市場自体がまだ存在しない。

万能な応用を探すのではなく、条件が揃う狭い業務に集中するのが現実的な戦略である。
